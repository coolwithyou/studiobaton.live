# Prep 행동 규칙

이슈 분석 요청을 받으면 다음 프로세스를 따르세요.

---

## Step 1: 이슈 분석

1. 이슈 본문에서 요구사항을 명확하게 추출하세요
2. 코드베이스를 탐색하여 관련 파일과 코드를 식별하세요
3. 변경이 미치는 영향 범위를 파악하세요

---

## Step 2: 분석 결과 코멘트 작성

아래 형식을 정확히 따라 코멘트를 작성하세요:

```
## 이슈 분석 결과

### 관련 파일
- `파일/경로.ts` - 해당 파일에서 수정할 내용 한 줄 요약

### 수정 플랜
1. 첫 번째 수정 단계 - 구체적인 변경 사항
2. 두 번째 수정 단계 - 구체적인 변경 사항
3. (필요한 만큼 추가)

### 예상 소요 시간
- 간단(~15분): 1-2개 파일, 단순 수정
- 보통(~1시간): 3-5개 파일, 로직 변경
- 복잡(반나절 이상): 다수 파일, 아키텍처 변경

### Confidence 평가
- HIGH: 수정 범위가 명확하고 1-2개 파일만 수정. 사이드이펙트 없음
- MEDIUM: 수정 가능하나 여러 파일 수정 또는 확인 필요
- LOW: 요구사항이 모호하거나 아키텍처 변경 필요
```

---

## Step 3: Confidence에 따른 행동 결정

### HIGH confidence인 경우
자동으로 수정을 진행하세요:
1. 이슈 번호 기반으로 브랜치 생성 (예: `fix/issue-42-간단한설명`)
2. 수정 플랜에 따라 코드 변경 수행
3. PR 생성 (이슈 번호 참조 포함)
4. 이슈에 PR 링크를 포함한 코멘트 작성

PR 본문 형식:
```
## Summary
이슈 #번호 에서 요청한 내용을 수정합니다.

## Changes
- 변경 사항 1
- 변경 사항 2

## Related Issue
Closes #번호
```

### MEDIUM 또는 LOW confidence인 경우
플랜만 코멘트로 남기세요. 자동 수정을 시도하지 마세요.
반드시 Claude Code 프롬프트를 포함하세요 (Step 4 참조).

---

## Step 4: Claude Code 프롬프트 생성

MEDIUM 또는 LOW confidence일 경우, 개발자가 Claude Code에 바로 붙여넣을 수 있는
구조화된 프롬프트를 코멘트에 포함하세요.

형식:
```
<details>
<summary>Claude Code 프롬프트 (복사해서 사용)</summary>

다음 이슈를 수정해주세요: #{이슈번호}

## 요구사항
- (이슈에서 추출한 요구사항)

## 수정 대상 파일
- `파일/경로` - 수정 내용

## 수정 방법
1. 단계별 구체적인 수정 지시
2. ...

## 주의사항
- 사이드이펙트 가능성
- 테스트 필요 사항

</details>
```

---

## Confidence 판단 기준

| 기준 | HIGH | MEDIUM | LOW |
|------|------|--------|-----|
| 수정 파일 수 | 1-2개 | 3-5개 | 6개 이상 |
| 변경 유형 | 텍스트, 스타일, 단순 로직 | 비즈니스 로직, API 수정 | 아키텍처, DB 스키마 |
| 요구사항 명확도 | 구체적 | 대체로 명확 | 모호/해석 필요 |
| 사이드이펙트 | 없음 | 제한적 | 광범위 |
| 테스트 필요성 | 기존 테스트 충분 | 일부 테스트 추가 | 대규모 테스트 필요 |

---

## 중요 규칙

1. **항상 코드를 먼저 읽으세요.** 추측하지 마세요.
2. **기존 코드 스타일을 따르세요.** 새로운 패턴을 도입하지 마세요.
3. **최소한의 변경만 하세요.** 요청받지 않은 리팩토링을 하지 마세요.
4. **HIGH confidence가 아니면 자동 수정하지 마세요.**
5. **PR 생성 시 이슈 번호를 반드시 참조하세요.**
